## 集成全新OCPI数据模块指南

本指南旨在为开发者提供详细的步骤，以在现有系统中集成全新的OCPI（Open Charge Point Interface）数据模块，例如`ChargingProfiles`。通过遵循以下流程，可以确保新模块的正确实现、版本兼容性和用户界面的一致性。

该系统采用基于Zod的JSON模式验证机制，支持多个OCPI版本（2.1.1-d2, 2.2.1-d2, 2.3.0），并根据所选版本动态加载相应的验证器和示例数据。新增模块需要在验证层、数据层和UI层进行同步更新。

## 关键步骤详解

### 步骤一：在ocpi-validators.js中创建Zod Schema定义

首先，需要为新的OCPI模块（如`ChargingProfiles`）定义一个Zod Schema。此Schema将用于验证传入的JSON数据是否符合OCPI规范。

应在`ocpi-validators.js`文件中，参照现有模块（如`LocationSchema_230`, `SessionSchema_230`）的格式，创建新的Schema常量。Schema应精确反映OCPI规范中该模块的数据结构、字段类型、枚举值和约束条件。

```javascript
// 示例：ChargingProfileSchema_230 的占位符定义
export const ChargingProfileSchema_230 = z.object({
    // ... 根据OCPI 2.3.0规范定义字段
});
```

### 步骤二：将新模块注册到各版本的ModuleValidators对象中

创建Schema后，必须将其注册到对应OCPI版本的`ModuleValidators`对象中，以便`validateOCPIJson`函数能够找到并使用它。

`ModuleValidators`是一个映射对象，其键为模块名称（如`locations`, `sessions`），值为对应的Zod Schema。需要检查`ModuleValidators_211`、`ModuleValidators_221`和`ModuleValidators_230`这三个对象，并根据新模块的版本支持情况决定注册位置。

例如，如果`ChargingProfiles`仅在OCPI 2.3.0中可用，则只需将其添加到`ModuleValidators_230`中。

```javascript
// 在 ocpi-validators.js 中更新 ModuleValidators_230
export const ModuleValidators_230 = {
    locations: LocationSchema_230,
    sessions: SessionSchema_230,
    bookings: BookingSchema_230,
    charging_profiles: ChargingProfileSchema_230, // 新增此项
    // ... 其他模块
};
```

### 步骤三：更新App.js的sampleDataMap和getVersionSpecificSampleData函数

为了使前端UI能够加载新模块的示例数据，需要在`App.js`中进行两项更新：

1.  **更新 `sampleDataMap`**: 这是一个用于向后兼容的旧版数据映射。虽然主要逻辑已迁移到版本特定的数据，但为保持一致性，建议在此处也添加新模块的入口。
2.  **更新 `getVersionSpecificSampleData` 函数**: 这是核心逻辑，用于根据当前选择的模块和版本返回正确的示例数据。需要在此函数中添加对新模块的支持，并判断其版本可用性。

```javascript
// 在 App.js 中更新 getVersionSpecificSampleData 函数
const getVersionSpecificSampleData = (module, version) => {
    const isVersion230 = version === '2.3.0';
    const isVersion211 = version === '2.1.1-d2';

    // ... 现有逻辑 ...

    if (module === 'charging_profiles') {
        // 假设该模块仅在2.3.0中可用
        return isVersion230 ? sampleData_230.charging_profile : null;
    }

    // ... 其他逻辑 ...
};
```

同时，确保`sampleDataMap`也包含一个指向示例数据的键。

```javascript
// 在 App.js 中更新 sampleDataMap
const sampleDataMap = {
    'locations': sampleLocation,
    'sessions': sampleSession,
    'charging_profiles': sampleChargingProfile, // 新增此项
    // ... 其他模块
};
```

### 步骤四：在UI Select下拉菜单中添加相应选项项

最后，为了让用户能够在界面上选择新模块，需要在`App.js`的UI组件中，于“选择模块”的下拉菜单（`<Select>`）内添加一个新的菜单项（`<MenuItem>`）。

这通常位于JSX的渲染部分。添加时，应考虑模块的版本限制，使用条件渲染来确保只有在支持的版本下才显示该选项。

```jsx
{/* 在 App.js 的 JSX 中 */}
<Select value={module} label="选择模块" onChange={(e) => setModule(e.target.value)}>
    <MenuItem value="locations">Locations</MenuItem>
    <MenuItem value="sessions">Sessions</MenuItem>
    {/* ... 其他现有选项 ... */}
    
    {/* 仅当版本为2.3.0时显示Charging Profiles选项 */}
    {version === '2.3.0' && (
        <MenuItem value="charging_profiles">Charging Profiles (2.3.0)</MenuItem>
    )}
</Select>
```

## 权限控制与版本兼容性

系统内置了严格的版本兼容性检查。`validateOCPIJson`函数会根据传入的`version`参数选择正确的验证器集。如果请求的模块在指定版本中不可用，函数会立即返回错误信息。

例如，`bookings`模块在`ModuleValidators_221`中不存在，因此当用户选择`2.2.1-d2`版本并尝试验证`bookings`数据时，`validateOCPIJson`会返回“Bookings模块仅在OCPI 2.3.0版本中可用”的错误。

```mermaid

```

## 验证与测试

完成上述所有步骤后，请进行以下验证：
1.  **功能验证**: 在UI中选择新模块和兼容的OCPI版本，点击“加载示例数据”按钮，确认能成功加载预设的示例数据。
2.  **验证测试**: 输入符合规范的JSON数据，点击“验证”按钮，确认返回“✅ 验证通过！”。
3.  **错误处理**: 输入格式错误或缺少必填字段的JSON，确认能收到具体的错误信息。
4.  **版本隔离测试**: 切换到不支持该模块的OCPI版本，确认该模块选项从下拉菜单中消失，且无法通过API进行验证。

通过以上全面的集成和测试，即可确保新的OCPI数据模块被正确、安全地集成到系统中。
